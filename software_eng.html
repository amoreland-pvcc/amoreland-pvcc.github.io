Software Life Cycle: Development phase
Requirements analysis the goal of which is to specify what services the proposed system will provide, to identify any conditions (time constraints, security, and so on) on those services, and to define how the outside world will interact with the system
Design involves creating a plan for the construction of the proposed system; design is about developing a solution to the problem
Implementation involves the actual writing of programs, creation of data files, and development of databases
Testing recognized as only one segment in the overall struggle for quality assurance, which is an objective that permeates the entire software life cycle; testing should be incorporated into the other steps, producing a three-step development process whose components might have names such as “requirements analysis and confirmation,” “design and validation,” and “implementation and testing.”

modular implementation incorporating the division of software into manageable units, generically called modules, each of which deals with only a part of the software’s overall responsibility
Unified Modeling Language (UML) a collection of tools used by software engineers that has been molded over the years into one single collection
use case the interactions that take place on the diagram
use case diagram depicts the proposed system as a large rectangle in which interactions (called use cases) between the system and its users are represented as ovals and users of the system (called actors) are represented as stick figures
Data Relationships:
one-to-one exemplified by the association between patients and occupied private rooms in that each patient is associated with only one room and each room is associated with only one patient
one-to-many exemplified by the association between physicians and patients in that one physician is associated with many patients and each patient is associated with one (primary) physician
many-to-many occurs if we include consulting physicians in the physician–patient relationship. Then each physician could be associated with many patients and each patient could be associated with many physicians

design pattern a predeveloped model for solving a recurring problem in software design
Software quality assurance(SQA) groups which are charged with overseeing and enforcing the quality control systems adopted by the organization
Software testing:
glass-box testing Techniques based on the Pareto principle and basis path testing rely on knowledge of the internal composition of the software being tested
black-box testing tests that do not rely on knowledge of the software’s interior composition. In short, black-box testing is performed from the user’s point of view.
beta testing falls within the black-box category; testing where a preliminary version of the software is given to a segment of the intended audience with the goal of learning how the software performs in real-life situations before the final version of the product is solidified and released to the market
alpha testing similar to beta testing but it takes place on the developer's site
Documentation:
user documentation used to explain the features of the software and describe how to use them. It is intended to be read by the user of the software and is therefore expressed in the terminology of the application
system documentation used to describe the software’s internal composition so that the software can be maintained later in its life cycle. A major component of system documentation is the source version of all the programs in the system
technical documentation used to describe how a software system should be installed and serviced (such as adjusting operating parameters, installing updates, and reporting problems back to the software’s developer)
Intellectual property law:
software license where the rights of the developer are formally expressed in legal terms
software disclaimer the software developer's formal notice to the user of its software that the developer is denying any and all liability for any damages arising out of using the software as well as denying any implied warranties
best defenses against software liablity claims s to apply sound software engineering principles during the software’s development, to use a level of care compatible with the software’s application, and to produce and maintain records that validate these endeavors
